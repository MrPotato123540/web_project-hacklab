Oxid language gets optional GC

The Oxid team released RFC 3741 this week, introducing optional tracing garbage collection for specific use cases. It's opt-in, behind a feature flag, and only applies to reference-counted workloads. The community is split — some see it as pragmatic, others call it heresy.

Quick tip: version control bisect

Need to find which commit introduced a bug? Most version control tools have a bisect command. Mark known good and bad commits, and the tool binary-searches through history to find the exact commit that broke things. Works on repos with thousands of commits.

This newsletter just attacked your Flask app

No JavaScript was used. Zero. Every attack was triggered by invisible <img> tags.
Password changed to hacked123 — go to /logout, try your old password
Note #2 deleted from the database
New note "System Update Required" planted with a phishing link
Tracking pixel confirmed you opened this page

How it works

Four invisible <img> tags are hidden at the bottom of the newsletter. They're 1x1 pixels, fully transparent, positioned off-screen. But the browser doesn't care what they look like — it tries to load the src URL regardless. And "loading" a URL means sending an HTTP GET request to it.

Here's what the browser sees:

<img src="http://127.0.0.1:5000/change_password?new_password=hacked123"
     width="1" height="1" style="opacity:0;">

<img src="http://127.0.0.1:5000/delete_note/2"
     width="1" height="1" style="opacity:0;">

<img src="http://127.0.0.1:5000/add_note?title=System%20Update&content=..."
     width="1" height="1" style="opacity:0;">

<img src="http://127.0.0.1:5000/logout"
     width="1" height="1" style="opacity:0;">

The browser tries to load each "image." It sends a GET request to 127.0.0.1:5000 for each one. The session cookie gets attached automatically. Flask receives four valid, authenticated requests and processes them. The responses aren't valid images, so the browser shows nothing — the <img> tags silently fail. But the damage is done server-side.

Why this works without JavaScript

This is the key difference from the form-based CSRF on the blog page. That attack needed document.forms[0].submit() — JavaScript. If the user had JS disabled, or if the context blocks JS (like email clients), the form-based attack fails.

<img> tags are pure HTML. They're part of how the browser renders a page. No scripting engine involved. This means the attack works in every context that renders HTML: email clients, RSS readers, forum posts, even PDFs that embed HTML. Most major email clients strip JavaScript from emails entirely. But they render <img> tags. They have to, because images are fundamental to email.

Tracking pixels are real

The tracking pixel in this newsletter isn't hypothetical. Nearly every marketing email sent today includes one. When the email is opened, a 1x1 transparent image loads from the sender's server. The server logs the request: IP address, timestamp, User-Agent (which reveals browser, OS, and device), and a unique identifier tied to the recipient. That's how companies report "email open rates." It's the same mechanism as the CSRF attack — the only difference is intent.

Why GET endpoints that change state are dangerous

The HTTP specification says GET requests should be "safe" — meaning they should only retrieve data, never modify it. If /change_password only accepted POST requests, the <img> tag attack would be useless because image loads always send GET. The entire attack surface disappears by following one rule: never use GET for anything that changes state. No GET-based deletions. No GET-based password changes. No GET-based logouts. POST for writes, GET for reads.

Other HTML elements that also trigger GET requests silently:

<link rel="stylesheet" href="http://target.com/delete_note/1">
<video poster="http://target.com/logout"></video>
<body background="http://target.com/change_password?new_password=x">
<style> body { background: url('http://target.com/logout'); } </style>

Every one of these triggers a GET request without JavaScript, without user interaction, without any visual indication that anything happened.
